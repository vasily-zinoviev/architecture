# 🤖 Программа развития: AI-Driven Architecture and Development
**Для Full Stack Системного Аналитика**

**Цель:** Научить системного аналитика не просто «использовать ИИ», а проектировать архитектуру, где AI является первоклассным компонентом, оценивать риски, выбирать подходящие модели и интегрировать инструменты генеративного ИИ в процесс разработки и анализа.

---

## 📚 Раздел 1. Фундамент: Понимание возможностей и ограничений ИИ
*База: Различение между классической логикой и вероятностными моделями.*

### 1.1. Типы ИИ и их применимость в архитектуре
**Контрольные вопросы:**
1. В каких сценариях лучше использовать детерминированный код (алгоритмы), а когда необходимо применять ML-модель (вероятностный подход)?
2. В чем принципиальная разница между классическим API и API, работающим на основе LLM (Large Language Models)?
3. Как вы определите, что задача требует обучения модели (Training), а что можно решить с помощью Fine-tuning или RAG (Retrieval-Augmented Generation)?

**Критерии оценки (Ответы):**
| Уровень | Характеристика ответа |
| :--- | :--- |
| **Низкий** | Считает, что ИИ может решить любую задачу. Не видит разницы между правилом «если-то» и предсказанием. |
| **Средний** | Понимает базовые термины (NLP, Computer Vision). Знает, что LLM хороши для текста, но не понимает, когда RAG лучше, чем Fine-tuning. |
| **Высокий** | Четко разделяет задачи на детерминированные и вероятностные. Знает, что RAG подходит для работы с приватными данными без переобучения, а Fine-tuning — для стиля и специфической терминологии. Может аргументировать выбор архитектуры (Serverless vs GPU-кластер) под задачу. |

### 1.2. Ограничения и риски ИИ (Hallucinations, Bias, Latency)
**Контрольные вопросы:**
1. Как вы спроектируете систему, чтобы минимизировать риск галлюцинаций (выдумывания фактов) в ответах LLM?
2. Какие метрики вы будете использовать для оценки качества работы ИИ-компонента (не просто accuracy, а что еще)?
3. Как вы оцените влияние задержки (latency) генерации текста на пользовательский опыт (UX) и как это компенсируете?

**Критерии оценки (Ответы):**
| Уровень | Характеристика ответа |
| :--- | :--- |
| **Низкий** | Считает, что ИИ всегда прав. Не учитывает задержки и стоимость запросов. |
| **Средний** | Знает о галлюцинациях и предлагает «человека в контуре» (Human-in-the-loop) для проверки. Понимает, что LLM медленнее скрипта. |
| **Высокий** | Проектирует архитектуру с валидацией ответов (Self-Consistency, Chain of Verification). Предлагает стратегии кэширования, асинхронной обработки и прогрессивного рендеринга (Streaming). Включает в требования метрики: токсичность, релевантность, стоимость токена за ответ. |

## 🏗️ Раздел 2. Архитектура на основе ИИ (AI-Native Architecture)
*Ключевая компетенция: Проектирование системы, где ИИ — не плагин, а ядро.*

### 2.1. Паттерны интеграции ИИ: RAG, Agents, Fine-tuning
**Контрольные вопросы:**
1. Опишите архитектуру решения на базе RAG (Retrieval-Augmented Generation). Какие компоненты обязательны (Embedding, Vector DB, LLM)?
2. В чем разница между простой чат-бот-интерфейсом и многоагентной системой (Multi-Agent System)? Когда нужен второй подход?
3. Как вы спроектируете систему, где ИИ-агент должен выполнять действия в других сервисах (Action Planning)?

**Критерии оценки (Ответы):**
| Уровень | Характеристика ответа |
| :--- | :--- |
| **Низкий** | Описывает просто «чат с базой знаний». Не понимает, как агент может взаимодействовать с API других систем. |
| **Средний** | Знает компоненты RAG. Понимает, что агенты могут выполнять функции, но не видит сложностей с безопасностью и управлением состоянием. |
| **Высокий** | Проектирует полную цепочку: *Query -> Embedding -> Vector Search -> Prompt Engineering -> LLM -> Function Calling -> API Action*. Предусматривает механизмы валидации действий агента перед отправкой в продакшн (Safety Guardrails). |

### 2.2. Обработка данных и векторные базы данных (Vector DB)
**Контрольные вопросы:**
1. Как вы подготовите данные для векторизации (Data Preprocessing) для RAG-системы? (Чистка, сегментация, метаданные).
2. В чем разница между плотными векторами и гибридным поиском (Dense + Sparse search)?
3. Как вы обеспечите актуальность векторной базы при частом обновлении исходных данных?

**Критерии оценки (Ответы):**
| Уровень | Характеристика ответа |
| :--- | :--- |
| **Низкий** | Считает, что «загрузил файл — и он работает». Не учитывает потерю смысла при нарезке текста (Chunking). |
| **Средний** | Понимает важность Chunking и метаданных. Знает про Vector DB, но не понимает, как обновлять индексы. |
| **Высокий** | Проектирует стратегию Chunking (по смыслу, по размеру, с перекрытием). Предлагает гибридный поиск для точности. Внедряет пайплайны (ETL) для периодического переиндексирования и управления версионностью данных. |

## 🛠 Раздел 3. Инструментарий и AI-Driven Development (AI4AD)
*Применение ИИ для ускорения собственной работы аналитика и команды.*

### 3.1. Использование LLM для генерации артефактов
**Контрольные вопросы:**
1. Как вы сформулируете промпт (prompt engineering), чтобы получить из ИИ качественный User Story или Acceptance Criteria?
2. Как вы используете ИИ для генерации SQL-запросов, диаграмм (Mermaid/UML) или тестовых сценариев?
3. Как вы проверяете качество кода или требований, сгенерированных ИИ, и какие риски при этом возникают?

**Критерии оценки (Ответы):**
| Уровень | Характеристика ответа |
| :--- | :--- |
| **Низкий** | Использует ИИ как «поисковик». Печатает «напиши ТЗ» без контекста. Слепо доверяет результату. |
| **Средний** | Знает про контекст и итерации промптов. Использует ИИ для черновиков. Проверяет результат вручную. |
| **Высокий** | Владеет техниками: Few-Shot prompting, Chain-of-Thought, Role-playing. Создает собственные шаблоны (Prompt Libraries) для команды. Внедряет процессы автоматической валидации сгенерированного кода/требований (AI-as-a-Reviewer). |

### 3.2. AI-инструменты в процессе анализа (Copilot, DevTools)
**Контрольные вопросы:**
1. Какие инструменты (GitHub Copilot, Cursor, специализированные аналитические плагины) вы используете для ускорения работы?
2. Как вы используете ИИ для реверс-инжиниринга (Reverse Engineering) легаси-кода и восстановления документации?
3. Как вы оцениваете безопасность при использовании корпоративных данных в публичных LLM?

**Критерии оценки (Ответы):**
| Уровень | Характеристика ответа |
| :--- | :--- |
| **Низкий** | Не использует ИИ-инструменты или использует только для простых текстов. Не задумывается о безопасности. |
| **Средний** | Использует IDE-помощников. Знает про риски утечки данных, но не имеет строгих правил. |
| **Высокий** | Внедряет политику безопасности (Data Governance): использование только корпоративных моделей, анонимизация данных перед отправкой. Использует ИИ для глубокого анализа легаси-кода, автоматической генерации документации и поиска уязвимостей в логике. |

## ⚖️ Раздел 4. Этика, Безопасность и Управление данными (AI Governance)
*Ответственность за решения, принятые с помощью ИИ.*

### 4.1. Безопасность и приватность в ИИ-системах
**Контрольные вопросы:**
1. Как вы защитите систему от Prompt Injection (атак через промпты)?
2. Как вы обеспечите конфиденциальность данных пользователей при работе с внешними LLM?
3. Какие механизмы вы внедрите для предотвращения генерации токсичного или дискриминационного контента?

**Критерии оценки (Ответы):**
| Уровень | Характеристика ответа |
| :--- | :--- |
| **Низкий** | Считает, что «ИИ сам разберется». Не видит угроз инъекций. |
| **Средний** | Знает о рисках. Предлагает базовую фильтрацию ввода/вывода. Использует PII (Personally Identifiable Information) маскирование. |
| **Высокий** | Проектирует архитектуру с «Guardrails» (фильтры безопасности на уровне системы). Внедряет Sandboxing для агентов. Использует локальные модели для чувствительных данных. Предусматривает аудит логов промптов и ответов. |

### 4.2. Управление качеством данных (Data Quality for AI)
**Контрольные вопросы:**
1. Почему «Garbage In, Garbage Out» критично для ИИ-систем и как это влияет на требования к данным?
2. Как вы оцените качество данных для обучения или дообучения модели?
3. Как вы документироваете происхождение данных (Data Lineage) в контексте ИИ-решений?

**Критерии оценки (Ответы):**
| Уровень | Характеристика ответа |
| :--- | :--- |
| **Низкий** | Считает, что данные — это просто «сырье». Не требует специфических метрик качества. |
| **Средний** | Понимает важность чистоты данных. Требует проверки на дубликаты и ошибки. |
| **Высокий** | Вводит требования к полноте, точности, актуальности и сбалансированности данных (Bias check). Проектирует пайплайны Data Quality. Отслеживает Data Lineage от источника до векторной базы. |

## 🛠 Методика проверки компетенции (Практическое задание)

**Задание: «Проектирование ИИ-системы с нуля»**

1.  **Сценарий:** «Разработайте архитектуру системы поддержки клиентов для банка, которая умеет отвечать на вопросы по тарифам, проверять баланс (через API) и выявлять мошеннические операции».
2.  **Требования к аналитику:**
    *   Выбрать тип архитектуры (RAG vs Fine-tuning vs Agents).
    *   Описать компоненты (LLM, Vector DB, API Gateway, Safety Filters).
    *   Сформулировать критерии приемки (Accuracy, Latency, Safety).
    *   Предложить промпт-инжиниринговую стратегию.
    *   Описать риски (галлюцинации, утечка данных) и способы их mitigation.

**Критерии оценки:**
*   **Junior:** Предложит простого чат-бота. Не учтет безопасность и интеграцию с API.
*   **Middle:** Предложит RAG, учтет API. Опишет основные риски, но не детализирует защиту.
*   **Senior:** Предложит агента с валидацией действий. Опишет Guardrails, стратегию кэширования, обработку ошибок, метрики качества и план по Data Governance.

**Примечание ментора:**
AI-технологии развиваются стремительно. Ваша задача как аналитика — не знать все модели наизусть, а уметь быстро оценить, какая технология подходит под задачу бизнеса, и спроектировать гибкую архитектуру, которую легко будет менять, когда появятся новые инструменты. Фокус на **принципах**, а не на конкретных библиотеках.
